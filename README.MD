
1. Introduction
	UART-IP-Core is a configurable Universal Asynchronous Receiver/Transmitter (UART) intellectual property (IP) core written entirely in VHDL 
	for Intel® FPGA devices. The design has been developed with modularity, configurability, and reusability as primary objectives, 
	making it suitable for both academic research and industrial FPGA developments.

	The IP integrates independent transmit and receive FIFOs, a configurable baud-rate generator, optional parity support, configurable stop bits, 
	and a complete verification environment developed in ModelSim. The architecture is compatible with Intel Platform Designer and has been designed 
	to simplify integration into larger FPGA-based systems.

	Unlike minimal UART implementations intended only for demonstration purposes, this project focuses on clean RTL architecture, 
	parameterization through VHDL generics, portability, and ease of maintenance.

2. Why this IP?
	Universal Asynchronous Receiver/Transmitter (UART) remains one of the most widely used communication protocols in embedded systems 
	due to its simplicity, reliability, and low hardware requirements. Despite the availability of numerous open-source UART implementations, 
	many designs provide only basic functionality and offer limited configurability.

	This project was developed to provide a modular, reusable, and highly configurable UART IP Core suitable for FPGA-based systems. 
	The architecture emphasizes clean RTL design, parameterization through VHDL generics, and ease of integration into larger digital systems.

	Unlike minimal UART implementations intended solely for educational purposes, this IP includes practical features required in real-world 
	applications, such as configurable baud rate generation, independent transmit and receive FIFOs, optional parity support, configurable stop bits, 
	error detection, and Platform Designer compatibility.

	The project also serves as a demonstration of professional FPGA design practices, including modular architecture, verification through simulation, 
	and reusable hardware development.

	Whether used for learning, prototyping, or integration into larger FPGA projects, this IP Core provides a solid foundation for reliable UART 
	communication.

5. Architecture
	The transmitter module implements a finite state machine responsible for serializing parallel data into a standard UART frame. 
	The transmission sequence begins with the generation of a Start bit, followed by the configurable number of data bits, an optional parity bit, and one or two Stop bits depending on the selected configuration.

	Data are supplied by the transmit FIFO, allowing asynchronous communication with the user logic while minimizing processor latency and preventing data loss during burst transmissions.

    5.1 Baud Generator
		The Baud Generator is responsible for producing all timing signals required by the UART transmitter and receiver.
		It generates two independent timing pulses:
		- "baud_tick", which defines the UART transmission rate.
		- "sample16_tick", which operates at sixteen times the baud rate and is used by the receiver to perform 16× oversampling, improving synchronization accuracy and reception reliability.
		The module computes these timing signals from the system clock frequency (`clk_freq`) and the selected UART baud rate (`baud_rate`), both configurable through VHDL generics.

    


5.2 UART_TX
	The `UART_TX` module is responsible for converting parallel data into a standard UART serial frame. It retrieves data from the transmit FIFO 	whenever data are available and the transmitter is idle.

	The transmission process is controlled by a Finite State Machine (FSM), ensuring that each frame is transmitted according to the UART protocol.

	A complete UART frame consists of:
	- One Start bit
	- A configurable number of data bits
	- An optional parity bit
	- One or two Stop bits

	The transmitter operates using the `baud_tick` signal generated by the Baud Generator, guaranteeing accurate timing for each transmitted bit.

	When the transmission of a frame is completed, the module asserts the `TX_IRQ` signal to notify the surrounding logic that the transmitter is ready 		for a new data word.

	The modular architecture of the transmitter makes it fully configurable through VHDL generics, allowing the same RTL implementation to support 			different UART configurations without modifying the source code.
   
    5.3 UART_RX
	The `UART_RX` module is responsible for receiving serial data and converting it into parallel data according to the UART communication protocol.

	To improve reception reliability, the receiver uses the `sample16_tick` signal generated by the Baud Generator, implementing a **16× oversampling** 		technique. This method allows accurate detection of the Start bit and provides robust sampling even in the presence of small clock differences or 		signal noise.

	The reception process is controlled by a Finite State Machine (FSM). Once a valid Start bit is detected, the receiver samples each incoming data bit 	at the center of its bit period. After all data bits have been received, the module optionally checks the parity bit (if enabled) and finally 		verifies the Stop bit.

	If the received frame is valid, the parallel data are written into the receive FIFO, making them available to the user logic.

	The receiver also performs error detection and generates dedicated status signals for the following conditions:
	- **Parity Error** – Indicates that the received parity bit does not match the calculated parity.
	- **Framing Error** – Indicates that the expected Stop bit is invalid.
	- **Overrun Error** – Indicates that a new frame has been received while the previous data have not yet been read from the receive FIFO.

	The modular design of the receiver allows it to support different UART configurations through VHDL generics without requiring any modification to the RTL implementation.
   
    5.4 TX FIFO
	The Transmit FIFO (`TX FIFO`) is a First-In First-Out memory buffer placed between the user logic and the UART transmitter. Its primary purpose is 		to temporarily store outgoing data before they are transmitted serially.

	By buffering data, the TX FIFO decouples the operating speed of the user logic from the UART transmission rate. This allows the system to write 		multiple data words without waiting for the completion of each serial transmission, significantly improving communication efficiency.

	Whenever the UART transmitter becomes available, it automatically reads the next data word from the TX FIFO and starts a new transmission. This mechanism ensures continuous data flow while preventing data loss caused by timing differences between the processor and the UART transmitter.

	The FIFO depth is configurable through the `fifo_size` generic, allowing the memory capacity to be adapted to different application requirements.

	The module also provides status signals to simplify flow control:
	- **full** – Indicates that the FIFO is full and cannot accept additional data.
	- **empty** – Indicates that the FIFO contains no data available for transmission.

	The FIFO is implemented using synchronous logic and is fully synthesizable, making it suitable for both educational and industrial FPGA designs.
    
    5.5 RX FIFO
	The Receive FIFO (`RX FIFO`) is a First-In First-Out memory buffer used to store incoming data received by the UART receiver before they are 		processed by the user logic.

	The RX FIFO provides temporary storage for received data, preventing data loss when the user logic is unable to read data immediately after 		reception. This buffering mechanism is particularly useful when multiple UART frames are received in rapid succession.

	Whenever the UART receiver successfully validates a frame, the received data are automatically written into the RX FIFO. The user logic can then 		retrieve the stored data at its own pace through the FIFO read interface.

	The FIFO depth is configurable through the `fifo_size` generic, allowing the buffer capacity to be adjusted according to system requirements.

	To simplify data management, the module provides status signals indicating its current state:

	- **empty** – Indicates that no data are available for reading.
	- **full** – Indicates that the FIFO has reached its maximum storage capacity.

	The FIFO operation is controlled by independent write (`wr_en`) and read (`rd_en`) enable signals, ensuring reliable data transfer while preserving 		data integrity.

	Combined with the UART receiver, the RX FIFO enables robust and continuous reception of serial data, even when the user logic operates at a lower processing rate than the incoming communication stream.


6. Generic Parameters

                        (2, 4, 8, 16, 32, ...). 
data_bits   Positive    Defines the number of transmitted and received data bits per frame. Typical values 
                        are  5, 6, 7, 8, or 9 bits. 
stop_bits   Positive    Defines the number of stop bits transmitted after each frame. Supported values are 
                        1 or 2. 
parity_en   Boolean     Enables or disables parity generation and parity checking.

baud_rate   Integer     Defines the UART communication speed (baud rate). Typical values include
                        9600, 19200, 38400, 57600 and 115200 baud.

8. UART Frame Format

The UART transmitter serializes parallel data according to the standard UART protocol. Each transmitted frame consists of four fields transmitted sequentially from left to right.


+-----------+------------------+------------+-----------+
| Start Bit |   Data Bits      | Parity Bit | Stop Bit  |
+-----------+------------------+------------+-----------+
|     0     | 5...9 bits (LSB) | Optional   | 1 or 2    |
+-----------+------------------+------------+-----------+

Frame Fields

# Start Bit

Each transmission begins with a single **Start bit** (`0`). The falling edge of the Start bit informs the receiver that a new UART frame is about to be transmitted.

# Data Bits

The payload consists of a configurable number of data bits. The IP Core supports **5 to 9 data bits**, depending on the value of the `data_bits` generic.

Data are transmitted **Least Significant Bit (LSB) first**, which is the standard UART transmission order.

# Parity Bit

The parity bit is optional and is enabled by setting the `parity_en` generic.

When enabled, the parity bit provides a simple error detection mechanism by allowing the receiver to verify the integrity of the received data.

# Stop Bit(s)

The frame terminates with one or two Stop bits, according to the `stop_bits` generic.

The Stop bit indicates the end of the frame and allows the receiver to synchronize before the next transmission begins.

9. Internal Operation
	
The UART IP Core has been designed using a modular architecture in which each hardware block performs a dedicated function. Data transmission and reception are handled independently, allowing full-duplex communication while maintaining a simple and scalable design.

### Transmission Path

The transmission process begins when the user logic writes a parallel data word into the Transmit FIFO by asserting the `wr_en` signal.

Once data become available and the UART transmitter is idle, the `UART_TX` module automatically reads the next data word from the TX FIFO.

The transmitter serializes the parallel data into a standard UART frame composed of:

- Start bit
- Configurable data bits
- Optional parity bit
- Configurable stop bit(s)

Each transmitted bit is synchronized using the `baud_tick` signal generated by the Baud Generator.

After the complete frame has been transmitted, the `TX_IRQ` signal is asserted to indicate that the transmitter is ready for the next transmission.

### Reception Path

The reception process continuously monitors the `rx_pin` input for the detection of a valid Start bit.

Once a Start bit is detected, the receiver samples the incoming serial data using the `sample16_tick` signal, implementing a 16× oversampling technique to improve synchronization accuracy.

The received bits are converted into a parallel data word before optional parity verification and stop-bit validation are performed.

If no communication errors are detected, the received data are automatically written into the Receive FIFO.

The user logic can retrieve the received data by asserting the `rd_en` signal.

### Error Detection

The receiver continuously verifies the integrity of each received frame.

Three independent error indicators are provided:

- **Parity Error** – Generated when the received parity bit does not match the calculated parity.
- **Framing Error** – Generated when the Stop bit is invalid.
- **Overrun Error** – Generated when a new frame is received before the previous data have been read from the RX FIFO.

These status signals allow the user logic to detect communication faults and implement appropriate recovery mechanisms.

14. Timing

The UART IP Core operates synchronously with the system clock while generating the required baud rate internally through the Baud Generator.

The transmitter shifts one bit for each `baud_tick`, whereas the receiver samples incoming data using a **16× oversampling** technique driven by the `sample16_tick` signal. This approach improves synchronization accuracy and increases robustness against clock mismatch and signal noise.

### Timing Characteristics

| Parameter     | Description                          |
|---------------|--------------------------------------|
| System Clock  | User-configurable (`clk_freq`)       |
| Baud Rate     | User-configurable (`baud_rate`)      |
| TX Bit Timing | One bit transmitted per `baud_tick`  |
| RX Sampling   | 16 samples per bit (`sample16_tick`) |
| Start Bit     | 1 bit                                |
| Data Bits     | Configurable (5–9 bits)              |
| Parity Bit    | Optional                             |

15. How to Use?

This section describes the basic steps required to integrate and use the UART IP Core in an FPGA project.

## Step 1 – Add the IP Core

Copy the UART IP Core source files into your Quartus project or include the generated Platform Designer component.

The required files include:

- `SYS_TOP.vhd`
- `UART_TX.vhd`
- `UART_RX.vhd`
- `FIFO.vhd`
- `Baud_Generator.vhd`
- `my_package.vhd`


## Step 2 – Configure the Generic Parameters
Adjust the generic parameters according to the application requirements.

```vhdl
generic map (
    fifo_size => 16,
    data_bits => 8,
    stop_bits => 1,
    parity_en => FALSE,
    clk_freq  => 50000000,
    baud_rate => 9600)



## Step 3 – Connect the External Signals

Connect the UART IP Core to the FPGA system.

| Signal     | Description            |
|------------|------------------------|
| `clk`      | System clock           |
| `reset`    | Active-high reset      |
| `data_in`  | Parallel transmit data |
| `wr_en`    | TX FIFO write enable   |
| `data_out` | Parallel received data |
| `rd_en`    | RX FIFO read enable    |
| `tx_pin`   | UART transmit pin      |
| `rx_pin`   | UART receive pin       |

---

## Step 4 – Transmit Data

To transmit data:

1. Wait until the TX FIFO is not full (`full = '0'`).
2. Place the data on `data_in`.
3. Assert `wr_en` for one clock cycle.
4. Wait for the `TX_IRQ` signal indicating that the transmission has completed.

---

## Step 5 – Receive Data

To receive data:

1. Wait until the RX FIFO is not empty (`empty = '0'`).
2. Assert `rd_en` for one clock cycle.
3. Read the received byte from `data_out`.

---

## Step 6 – Error Handling

The UART receiver provides three independent error flags:

- `parity_error`
- `framing_error`
- `overrun_error`

These signals should be monitored by the user logic to detect communication faults.

## Step 7 – Platform Designer Integration

The UART IP Core can also be instantiated as a custom Intel® Platform Designer component.

After adding the generated component to the IP Catalog:

1. Instantiate the component.
2. Connect the Clock and Reset interfaces.
3. Export the UART conduit interface.
4. Configure the generic parameters.
5. Generate the Platform Designer system.
6. Compile the Quartus project.


## 16. Future Improvements

Although the current UART IP Core provides a complete and configurable solution for reliable serial communication, several enhancements are planned to further improve its functionality, flexibility, and industrial applicability.

The following features are considered for future releases:

- **Hardware Flow Control**

- **Performance Optimization**

These improvements aim to extend the versatility of the UART IP Core while maintaining its modular architecture and ease of integration.
